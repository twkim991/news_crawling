# 아키텍처/파이프라인 점검 리포트

## 1) 현재 구조 요약
- `ag_pipeline.py`: AG News(4-class)에서 Sci/Tech vs Non-Tech 이진분류기 학습.
- `newsapi_pipeline.py`: NewsAPI 수집/정규화/전처리.
- `pipeline.py`: 운영 데이터(NewsAPI/SSAFY)만 대상으로 이진분류 -> 테크 뉴스 추출 -> 세부 카테고리 분류 -> 주간/월간 트렌드 리포트와 메타데이터 생성.
- `common.py`: 스키마 보정, 전처리, 임베딩 인코딩, 세부 카테고리(유사도 기반) 분류 로직.
- `classifier.py`: 이진분류기 로드/추론 로직.
- `analytics.py`: 실행 메타데이터와 주간/월간 트렌드 집계 생성.

## 2) 목표 대비 구조 효율성 평가

### 장점
- 학습(AG)과 운영 추론(NewsAPI/SSAFY)이 분리되어 실제 트렌드 분석 대상 데이터셋 경계가 더 명확해졌습니다.
- AG로 이진 분류기를 만들고, 그 결과에 대해 제로샷 성격의 세부 분류를 적용하는 2단계 구조는 목적(기술 스택 트렌드 분석)에 맞는 합리적 설계입니다.
- 실행 메타데이터와 주간/월간 카테고리 트렌드가 자동 생성되어 실험 재현성과 분석 활용도가 개선됐습니다.

### 남아있는 병목/리스크
1. **운영 평가셋 부족**
   - AG 외 실제 뉴스 샘플 라벨셋이 아직 없어 운영 정밀도/재현율을 객관적으로 보기 어렵습니다.
2. **세부 분류가 단일 대표 카테고리 중심**
   - 복합 기사(예: AI + Cloud + Product launch)에서 다중 태깅이 필요할 수 있습니다.
3. **수집 소스 편향 모니터링은 시작 단계**
   - 메타데이터에 소스 분포는 남기지만 source quality score나 편향 보정은 아직 없습니다.
4. **GDELT 등 대규모 외부 소스 미연동**
   - 글로벌 이벤트/정책 신호를 더 넓게 보기 위한 소스 확장이 남아있습니다.

## 3) 즉시 적용된 개선안
1. **운영 추론 데이터셋 분리**
   - `pipeline.py`는 더 이상 AG 학습 데이터를 합치지 않고 운영용 데이터(NewsAPI/SSAFY)만 처리합니다.
2. **공통 스키마 보정 추가**
   - `common.ensure_schema`로 필수 컬럼을 보정해 데이터 계약을 더 명시적으로 만들었습니다.
3. **실행 메타데이터 저장**
   - 입력 소스, 모델 경로, threshold, row 수, 카테고리 분포를 `outputs/metadata.json`에 기록합니다.
4. **트렌드 산출물 자동 생성**
   - `outputs/tech_weekly_trends.csv`, `outputs/tech_monthly_trends.csv`를 생성합니다.

## 4) 다음 단계 제안
1. **운영 평가셋 확장**
   - 실제 뉴스 샘플 라벨링 300~1000건 구축 후 monthly 회귀테스트.
2. **다중 태그/엔티티 추출 도입**
   - 대표 카테고리 + 보조 카테고리 + vendor/entity 추출 구조로 확장.
3. **설정 분리**
   - 쿼리, 날짜범위, 임계치, 모델경로를 YAML/ENV로 분리.
4. **GDELT 연동**
   - 이벤트성 글로벌 뉴스 수집을 위한 별도 ingestion pipeline 추가.

## 5) 결론
현재 구조는 **프로토타입을 넘어 운영 분석 파이프라인으로 가기 위한 최소 요건**을 일부 충족하기 시작했습니다.
특히
- 운영 데이터 분리,
- 실행 메타데이터,
- 주간/월간 트렌드 리포트
가 추가되면서 “기술 스택 뉴스 트렌드 파악”이라는 목적과의 정합성이 더 높아졌습니다.
다만 장기적으로는 운영 평가셋, 다중 태깅, 소스 확장이 핵심 과제로 남아 있습니다.
