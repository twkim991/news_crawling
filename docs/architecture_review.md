# 아키텍처/파이프라인 점검 리포트

## 1) 현재 구조 요약
- `ag_pipeline.py`: AG News(4-class)에서 Sci/Tech vs Non-Tech 이진분류기 학습.
- `newsapi_pipeline.py`: NewsAPI 수집/정규화/전처리.
- `pipeline.py`: AG + NewsAPI 병합 후 이진분류 -> 테크 뉴스만 추출 -> 세부 카테고리 분류.
- `common.py`: 전처리, 임베딩 인코딩, 세부 카테고리(유사도 기반) 분류 로직.
- `classifier.py`: 이진분류기 로드/추론 로직.

## 2) 목표 대비 구조 효율성 평가

### 장점
- 학습(AG)과 수집(NewsAPI), 추론(통합 파이프라인)이 파일 단위로 분리되어 있어 입문/실험 단계로는 이해가 쉽습니다.
- AG로 이진 분류기를 만들고, 그 결과에 대해 제로샷 성격의 세부 분류를 적용하는 2단계 구조는 목적(취업 준비생용 기술 트렌드 분석)에 맞는 합리적 설계입니다.

### 현재 병목/리스크
1. **중복된 분류 로직/모델 로딩 경로가 혼재**
   - `common.py`와 `classifier.py`에 임베딩 기반 기능이 분산되어 확장 시 유지보수 비용 증가.
2. **학습/추론 파이프라인의 데이터 스키마 계약(contract)이 명시적이지 않음**
   - 필요한 컬럼(`title`, `description`, `content`, `text`, `source`)이 코드 관례에 의존.
3. **실험 재현성 지표 부족**
   - 모델 버전, 데이터 기간, 파라미터, 임계치 변화가 메타데이터로 체계적으로 남지 않음.
4. **운영 관점 품질지표 부족**
   - 분류 정확도 외에 드리프트(기간별 score 분포 변화), 카테고리 쏠림, 소스 편향 모니터링 부재.

## 3) 즉시 적용 권장안 (Low effort / High impact)
1. **공통 임베딩 경로 단일화**
   - 임베딩 생성은 `common.encode_texts`를 단일 경로로 사용해 모델 중복 초기화를 방지.
2. **분류 단계 일관성 확보**
   - 세부 카테고리는 점수/갭 임계치를 포함한 `common.classify_subcategory`를 기준으로 통일.
3. **데이터/모델 실행 메타데이터 남기기**
   - 실행 시점, 입력 파일 해시, 데이터 기간, 임계치, 결과 row 수를 `outputs/metadata.json`에 누적.
4. **간단한 품질 게이트 도입**
   - 예: `tech_pred` 비율이 비정상(예: 5% 미만/95% 초과)일 때 경고 로그.

## 4) 중기 개선안
1. **패키지 구조 표준화 (`src/news_trend/..`)**
   - `training/`, `ingestion/`, `inference/`, `evaluation/`로 모듈 경계 명확화.
2. **설정 분리**
   - 쿼리, 날짜범위, 임계치, 모델경로를 코드 하드코딩 대신 YAML/ENV로 분리.
3. **평가셋 확장**
   - AG 외 실제 뉴스 샘플 라벨링 300~1000건 구축 후 monthly 회귀테스트.
4. **분석 목적 출력 강화**
   - 단순 CSV 외 주간/월간 카테고리 점유율, 신규 키워드, 소스별 편향 리포트 자동 생성.

## 5) 결론
현재 구조는 **프로토타입으로는 적절**하나, 서비스 목표(지속적 트렌드 분석/학습 우선순위 제시)를 위해서는
- 실행 재현성,
- 품질 모니터링,
- 설정/스키마 표준화
가 보강되어야 합니다.

이번 변경에서 우선 공통 임베딩 경로와 세부분류 경로를 단일화해, 향후 확장 시 유지보수 효율을 높이는 방향으로 정리했습니다.
