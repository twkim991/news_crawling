# GDELT source strategy

## 결론
이 프로젝트에는 **GDELT Context 2.0 API나 GDELT DOC API보다, GDELT 원시 배치 데이터 중 GKG(Global Knowledge Graph) 2.x 파일을 다운로드해서 사용하는 방식이 가장 적합**합니다.

## 비교 기준
이 프로젝트의 목적은 아래와 같습니다.
- 여러 뉴스 소스에서 기술 관련 뉴스만 선별
- 어떤 기술 스택/카테고리 뉴스인지 분류
- 주간/월간 트렌드 분석에 활용
- 특정 시점 샘플이 아니라 재현 가능한 데이터 파이프라인 구성

## 1. GDELT DOC API
### 장점
- 기사 URL, 제목/요약형 정보에 가까운 결과를 바로 얻기 쉽다.
- 빠르게 프로토타입을 만들기 좋다.
- 키워드 검색 기반으로 바로 기사 후보를 모을 수 있다.

### 단점
- 검색 API라서 **질의(query) 의존 샘플링**이 강하다.
- 대규모 기간 수집 시 `maxrecords` 제한과 구간 분할 전략이 필요하다.
- 프로젝트의 핵심인 **장기 추세 분석용 기준 데이터셋**으로 쓰기에는 누락/편향 관리가 어렵다.

### 이 프로젝트와의 적합성
- 빠른 데모/탐색용으로는 괜찮다.
- 하지만 “기술 뉴스 트렌드”를 안정적으로 비교하려는 현재 목적에는 **보조 수집 경로**에 가깝다.

## 2. GDELT Context 2.0 API
### 장점
- 문맥 단위 검색이 가능해서 특정 키워드가 실제로 어떤 문맥에서 쓰였는지 확인하기 좋다.
- 실시간 모니터링/탐색 UX에는 유용하다.

### 단점
- GDELT 공식 안내 기준으로 **검색 시계열 범위가 최대 72시간** 수준이다.
- 문서 전체가 아니라 매칭 문장 중심이라, 기사 단위 분류/집계 기준 데이터로 쓰기 불리하다.
- 주간/월간 트렌드용 백필(backfill) 데이터 소스로는 부적합하다.

### 이 프로젝트와의 적합성
- 실시간 탐색, 알림, 모니터링 UI용 보조 API로는 적합하다.
- 하지만 현재처럼 **월 단위 이상 트렌드 분석**을 해야 하는 프로젝트에는 맞지 않는다.

## 3. GDELT 원시 데이터 다운로드 (선택)
### 장점
- GDELT가 배치로 공개하는 원시 파일을 직접 내려받아 **재현 가능한 데이터셋**을 만들 수 있다.
- 기간을 넓게 잡아도 API 결과 제한에 덜 묶인다.
- 기사 메타데이터(테마, 인물, 기관, 위치, 톤, 문서 식별자 등)를 구조적으로 활용할 수 있다.
- 주간/월간/분기 단위 트렌드 분석처럼 **시계열 누적 분석**에 가장 잘 맞는다.

### 단점
- 파싱/정제 로직을 직접 유지해야 한다.
- DOC API보다 바로 읽기 쉬운 title/summary가 부족할 수 있다.
- 저장량과 처리량 관리가 필요하다.

### 이 프로젝트와의 적합성
- 현재 목표와 가장 잘 맞는다.
- 특히 **GKG 2.x**는 기사별 테마/기관/인물/톤 정보를 포함해 기술 키워드 후보 추출과 트렌드 집계에 유리하다.

## 최종 선택
- **기본 수집 소스는 GDELT raw GKG 2.x 파일 다운로드 방식으로 결정**합니다.
- **DOC API는 탐색/샘플링/개발 초기 검증용 보조 경로**로만 고려합니다.
- **Context 2.0 API는 실시간 모니터링 보조 기능이 필요할 때만 별도 사용**하는 것이 적절합니다.

## 코드 반영 원칙
- `src/gdelt_pipeline.py`는 DOC API가 아니라 **`masterfilelist.txt` 기반 raw GKG 다운로드 파이프라인**으로 구현합니다.
- GKG의 `themes`, `organizations`, `persons`, `locations`, `tone`, `document_identifier`를 공통 스키마로 정규화합니다.
- 기술 뉴스 후보 필터링은 GKG 메타데이터 기반 keyword filtering으로 1차 선별한 뒤, 기존 분류/트렌드 파이프라인에 연결합니다.
